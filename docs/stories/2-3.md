# Story 2-3: `signer wrapper (G4) — falcon512paddedEth via fork injection`

> **Size history:** `plan.md` originally sized **L** (4 tasks; ground-up hybrid-fork sign state machine in our repo). A-005 forward contract resized to **M** (~285 LOC source-transplant of `signRaw` + `FFSampler` + `HashToPoint` + `completePrivate` into `test/signers/falcon-eth.core.ts`, gated through `__tests` escape-hatch imports). **Final design (A-006 — Strategy E):** **S / 4 tasks / ~18 LOC in the fork + ~250 LOC in our repo.** The fork at `github.com/LimeChain/noble-post-quantum-eth` (private) adds a factory-level `hashToPoint` injection seam to noble's `genFalcon`. Our repo consumes the fork via a `git+ssh://` branch pin and wraps `falcon512paddedEth.sign(...)` + `encodeSignatureForZKNOX` (existing Story 2-2 helper). **No TS-side port of signRaw / FFSampler / HashToPoint / completePrivate; no `__tests` dependency.** See `docs/amendments.md` §A-006 for full rationale + fork audit surface.

## Why This Story Exists

**G4 is the signer byte-identity gate for the falcon-eth oracle chain.** Downstream Story 2-4 (integration + G5 pk-transform + G6 on-chain validateUserOp) consumes G4's signer surface; a byte-wrong signer poisons every downstream assertion.

ETHFALCON's `.rsp` signatures were generated by Python's `sk.sign(msg, xof=KeccakPRNG)` using a DRBG-seeded `randombytes` stream (`ETHFALCON/pythonref/test_falcon_KAT.py:134`). Strategy E's insight, verified across two 2026-04-20 spikes:

1. **Noble's `signRaw` + `FFSampler` + `completePrivate` implement the Falcon reference algorithm verbatim, EXCEPT for `HashToPoint`'s XOF** — noble hardcodes `shake256.create().update(nonce).update(msg)` at the HashToPoint inner function (`src/falcon.ts:1752-1779`); ETHFALCON uses `KeccakPRNG`-driven HashToPoint, byte-compatible with our Story 2-2 Solidity-anchored `hashToPointEVM`.
2. **HashToPoint is a factory-scope closure binding, not a closure-private helper reached via `__tests`.** Replacing its definition inside `genFalcon` requires no change to `signRaw` (line 2191) or `verifyRaw` (line 2274) call sites — they resolve `HashToPoint` by name against whichever definition is bound at factory-construction time.

Design B (factory-level injection) is therefore a ~17-LOC diff on the fork's `src/falcon.ts` plus `"prepare": "tsc"` in the fork's `package.json` (noble gitignores compiled dist; git-dep consumers must rebuild). Our repo's addition is a three-line `falcon512paddedEth = genFalcon({ ...falcon512paddedOpts, hashToPoint: hashToPointEVM })` inside `test/signers/falcon-eth.core.ts` (which already owns `hashToPointEVM` from Story 2-2). `falcon512paddedOpts` is a new fork-side constant that factors out noble's inline `{ ...falcon512opts, padded: true, maxS2Len: 625 }` at the `falcon512padded` construction — exported so our repo inherits ETHFALCON-compatible bounds implicitly (no magic numbers).

The encoding delta between noble's ~666 B detached output and the 1064 B `salt(40) || s2_compact(1024)` fixture layout (Story 1-1 captured this shape from ETHFALCON's Python sign + falcon_compact packing at `scripts/generate-kat-fixtures.ts:1310-1337`) is already handled by `encodeSignatureForZKNOX` at `test/signers/falcon-encoding.ts:159` — built for Story 2-2 and landed pre-2-3. Story 2-3 is therefore (a) a small fork change, (b) a `falcon512paddedEth` wiring, (c) thin wrapper files, (d) a G4 KAT test. No new cryptographic code, no `XofFactory` abstraction, no `signWithXofInstrumented`.

## User Story

As a signer-catalogue maintainer, I want `signWithKatBytes(sk, msg, reader)` to wrap `falcon512paddedEth.sign(...)` and produce byte-identical 1064 B signatures to ETHFALCON's `.rsp` vectors over ≥100 vectors — without copy-porting noble's sign pipeline into our repo, without depending on noble's `__tests` escape hatch, and without maintaining a drift-prone delta enumeration.

### Our repo — `falcon512paddedEth` instantiation (+~10 LOC in `falcon-eth.core.ts`)

`test/signers/falcon-eth.core.ts` (Story 2-2 landed `hashToPointEVM` here) appends:

```ts
import { genFalcon, falcon512paddedOpts, type Falcon } from "@noble/post-quantum/falcon.js";

/**
 * Falcon-ETH variant of noble's `falcon512padded` — swaps the factory's
 * HashToPoint binding for our Story 2-2 `hashToPointEVM` (Keccak256-based,
 * byte-identical to ETHFALCON's Solidity + Python references per G2 KAT).
 *
 * The `falcon512paddedOpts` spread carries `padded: true` + `maxS2Len: 625`
 * from the fork — the ETHFALCON-compatible rejection bound
 * (`compress(s[1], 625)` at ETHFALCON/pythonref/falcon.py:474). Noble's
 * unpadded `falcon512` uses `maxS2Len: 711` — a different acceptance
 * criterion that produces different s2 polynomials on rejection iterations,
 * breaking G4 byte-identity. Consuming `falcon512paddedOpts` (the dedicated
 * fork-exported opts bundle) makes this parity point implicit — no magic
 * numbers in our repo.
 */
export const falcon512paddedEth: Falcon = genFalcon({
  ...falcon512paddedOpts,
  hashToPoint: hashToPointEVM,
});
```

**Why we consume `falcon512paddedOpts` (not `falcon512opts`):** ETHFALCON Python's `sk.sign` at `ETHFALCON/pythonref/falcon.py:474` calls `compress(s[1], self.sig_bytelen - HEAD_LEN - SALT_LEN)` = `compress(s[1], 666 - 1 - 40)` = `compress(s[1], 625)`. The signing rejection loop retries if compression fails (s[1] doesn't fit in 625 B). Noble's `falcon512` (unpadded) uses `maxS2Len: 711` (see `src/falcon.ts:2425`); `falcon512padded` uses `maxS2Len: 625` + `padded: true` (line 2437-2441). For byte-identity against `.rsp` vectors, our fork's Eth variant MUST match the padded bounds. By consuming `falcon512paddedOpts` — the fork-exported opts bundle that factors out noble's inline padded construction — our repo inherits the ETHFALCON-compatible bounds implicitly. No hardcoded `625` or `padded: true` in our repo; no drift risk if noble ever re-tunes padded's opts. Gate 5's `grep "falcon512paddedOpts" test/signers/falcon-eth.core.ts` sanity check pins this.

### Signer wrappers — thin glue (no cryptographic logic)

```ts
// test/signers/falcon-eth.kat-internal.ts (T3 addition — existing file owns keygenInternal from Story 2-1)

import { hexToBytes } from "viem";

import { falcon512paddedEth } from "./falcon-eth.core.js";
import { encodeSignatureForZKNOX } from "./falcon-encoding.js";
import { SignerInputError } from "./errors.js";

/**
 * Sequential byte stream for deterministic KAT signing. Production `signUserOp`
 * allocates 88 B from `crypto.getRandomValues` and wraps it as a `BytesReader`;
 * G4 KAT wraps `rngAesCtrDrbg256` advanced past the keygen draw.
 */
export interface BytesReader {
  /** Returns exactly `n` bytes. Implementations enforce budgets at this boundary. */
  read(n: number): Uint8Array;
}

const SIGNING_BUDGET = 88; // 40 B salt + 48 B sampler seed (A-005 DRBG decomposition)
const SECRET_KEY_LEN = 1281;

/**
 * Deterministic Falcon-ETH sign. Returns 1064 B `salt(40) || s2_compact(1024)`
 * — the ETHFALCON on-chain format captured in `vectors.json#signature` fields.
 * Routes through `falcon512paddedEth.sign` (HashToPoint-injected fork; see
 * `falcon-eth.core.ts`) then re-encodes via `encodeSignatureForZKNOX`.
 *
 * @throws SignerInputError INVALID_SECRET_KEY_LENGTH / INVALID_MESSAGE /
 *         SIGNING_BYTES_EXHAUSTED
 */
export function signWithKatBytes(
  sk: Uint8Array,
  msg: Uint8Array,
  reader: BytesReader,
): Uint8Array {
  if (!(sk instanceof Uint8Array) || sk.length !== SECRET_KEY_LEN) {
    throw new SignerInputError(
      "INVALID_SECRET_KEY_LENGTH",
      `expected ${SECRET_KEY_LEN} B sk, got ${sk instanceof Uint8Array ? sk.length + " B" : typeof sk}`,
    );
  }
  if (!(msg instanceof Uint8Array) || msg.length === 0) {
    throw new SignerInputError(
      "INVALID_MESSAGE",
      `expected non-empty Uint8Array msg, got ${msg instanceof Uint8Array ? "length 0" : typeof msg}`,
    );
  }
  // Over-draw guard — noble's signRaw may call `random(40)` then `random(48)`
  // (ETHFALCON contract); a misbehaving reader that serves more signals
  // a port bug (KAT) or an exhausted source (production).
  let consumed = 0;
  const guardedRandom = (n?: number): Uint8Array => {
    const len = n ?? 0;
    if (consumed + len > SIGNING_BUDGET) {
      throw new SignerInputError(
        "SIGNING_BYTES_EXHAUSTED",
        `reader over-draw: ${consumed} + ${len} B > ${SIGNING_BUDGET} B budget`,
      );
    }
    consumed += len;
    return reader.read(len);
  };
  const nobleSig = falcon512paddedEth.sign(msg, sk, { random: guardedRandom });
  return hexToBytes(encodeSignatureForZKNOX(nobleSig));
}
```

For `signUserOp` (production surface in `falcon-eth.ts`): `computeUserOpHash` → allocate 88 B `crypto.getRandomValues` buffer → wrap as sequential `BytesReader` → call `signWithKatBytes` → pack result into `PackedUserOperation` with `bytesToHex(sig)`. Mirrors `test/signers/ml-dsa-eth.ts#signUserOp` structure one-for-one (AC-7 style uniformity).
