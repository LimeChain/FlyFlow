# Story 2-1: `keygen wrapper (G3) + fixture schema extension`

> **Size history:** originally L (5 tasks, ~500 LOC NTRU source-transplant) per story-creator's first draft. Resized to S after empirical byte-identity + architectural re-grounding (A-005). This story's T1 does the doc sweep + fixture schema extension that forward-binds Story 2-3 as well. Gate target is still G3 (keygen byte-identity). See `docs/amendments.md` §A-005 for the full rationale and the forward contract for Story 2-3.

## Why This Story Exists

**G3 is the keygen byte-identity gate.** Downstream Stories 2-3 (signer) and 2-4 (pk-transform + integration) require byte-identical keys to ETHFALCON's `.rsp` vectors so their G4/G5 assertions are not poisoned by upstream drift.

**Two verified facts (see A-005 §Evidence for the full proof chain):**

1. **ETHFALCON keygen ≡ NIST Falcon keygen, byte-for-byte.** Both share `AES256_CTR_DRBG → SHAKE256 → ntru_gen` (line-by-line identical in `test_falcon_KAT.py:80-89` and `:122-131`). Only signing XOF differs (SHAKE vs KeccakPRNG). Empirical TS byte-identity verified against noble 2026-04-20.
2. **DRBG already runs in Python at fixture-gen time.** `docs/architecture.md:33+90` binds this; `scripts/generate-kat-fixtures.ts:870-902` already invokes `AES256_CTR_DRBG(seed)` via `python3 -c` for falcon-eth's existing fixture fields. The pattern was carried from mldsa-eth per `docs/research.md:109`.

**Consequence:** Story 2-1 is (a) a Python-recorder extension that captures `innerSeed` + `signingDrbg` per vector into the fixture, (b) a thin TS wrapper that calls `noble.falcon512.keygen(innerSeed)`. Zero new cryptographic code in TS. The `signingDrbg` field is a deposit for Story 2-3 (signer consumes it via a `BytesReader`; also no TS-side DRBG).

## User Story

As a signer-catalogue maintainer, I want `keygenInternal(innerSeed)` to wrap noble's Falcon-512 keygen with the DRBG-derivation already done at fixture-gen time, so G3 passes byte-identity over ≥100 `.rsp` vectors without adding any cryptographic primitive to the TypeScript surface.

## Architecture Guardrails

### A-005 is authoritative — DRBG lives in Python

Per `docs/amendments.md` §A-005 + architecture `:33+90`: AES-256-CTR-DRBG runs in the Python fixture-gen subprocess, **not** in TypeScript. The fixture stores `innerSeed` (keygen's first draw) and `signingDrbg` (post-keygen draws). TS never instantiates a DRBG.

### Python recorder pattern (T1 inline Python sketch)

The Python recorder wraps the DRBG passively — it does NOT try to predict byte counts:

```python
class RecordingDRBG:
    def __init__(self, inner):
        self.inner = inner
        self.drawn = bytearray()
    def random_bytes(self, n):
        b = self.inner.random_bytes(n)
        self.drawn += b       # capture exactly what was returned
        return b

# Fixture-gen flow per vector (extended from existing lines ~880-895 in the
# inline Python source string of writeFalconEthFixture / runFalconPythonBatch):
drbg = AES256_CTR_DRBG(seed)
inner_seed = drbg.random_bytes(48)              # keygen draw — store as `innerSeed`
prng = SHAKE.new(inner_seed); prng.flip()
f, g, F, G = ntru_gen(512, randombytes=prng.read, logn=9)
sk = SecretKey(512, [f, g, F, G])
pk = PublicKey(512, sk.h)

rec = RecordingDRBG(drbg)                        # wrap same drbg (state continues)
sig = sk.sign(msg, randombytes=rec.random_bytes, xof=KeccakPRNG)
signing_drbg = rec.drawn                         # variable-length — captured, not guessed

# emit fields (JSON):
# {
#   "innerSeed":   inner_seed.hex(),
#   "signingDrbg": signing_drbg.hex(),
#   ...existing fields (pkCoeffsHex, signature, etc.)
# }
```

**Invariants to assert at T1:**
- `len(inner_seed) == 48` (exactly — AES-CTR-DRBG contract).
- `len(signing_drbg) > 0` (signing must draw at least the 40 B salt).
- `len(signing_drbg) % 1 == 0` (trivially — it's bytes, not bits).
- Across 100 vectors, log min/max/mean `len(signing_drbg)` as diagnostic. Expect 2-12 KB typical (Falcon-512 Gaussian-sampling rejection distribution). A single vector with length 10-100× the mean indicates a rejection-loop anomaly — investigate before committing.

### TS wrapper contracts (T2)

```ts
// test/signers/falcon-eth.kat-internal.ts (new, ~15 LOC — plus types + error class import)
import { falcon512 } from "@noble/post-quantum/falcon.js";
import { SignerInputError } from "./errors.js";  // or wherever the project's error type lives

export interface FalconEthKeypair {
  readonly publicKey: Uint8Array;   // 897 B
  readonly secretKey: Uint8Array;   // 1281 B
}

export function keygenInternal(innerSeed: Uint8Array): FalconEthKeypair {
  if (!(innerSeed instanceof Uint8Array) || innerSeed.length !== 48) {
    throw new SignerInputError(
      "INVALID_INNER_SEED_LENGTH",
      `expected 48 B Uint8Array, got ${innerSeed instanceof Uint8Array ? `${innerSeed.length} B` : `non-Uint8Array (${typeof innerSeed})`}`
    );
  }
  return falcon512.keygen(innerSeed);
}
```

```ts
// test/signers/falcon-eth.ts (new, ~15 LOC)
import { falcon512 } from "@noble/post-quantum/falcon.js";

export interface FalconEthKeypair {
  readonly publicKey: Uint8Array;
  readonly secretKey: Uint8Array;
}

export function keygen(): FalconEthKeypair {
  const seed = new Uint8Array(48);
  globalThis.crypto.getRandomValues(seed);
  return falcon512.keygen(seed);
}
```
