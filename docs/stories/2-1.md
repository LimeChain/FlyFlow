# Story 2-1: `keygen wrapper (G3) — noble.falcon512 + rngAesCtrDrbg256`

> **Size history:** originally L (5 tasks, ~500 LOC NTRU source-transplant) per story-creator's first draft. Briefly resized to S via a "fixture schema extension + Python recorder" design (A-005 v1). **Final design (A-005 v2):** S / 3 tasks / ~100 LOC. Library-first — `@noble/post-quantum/falcon.js#falcon512.keygen` for keygen math, `@noble/ciphers/aes.js#rngAesCtrDrbg256` for NIST-KAT DRBG replay at test time. **No fixture changes. No Python changes. No new crypto code.** Gate target: G3 (keygen byte-identity over ≥100 `.rsp` vectors). See `docs/amendments.md` §A-005 for the full rationale + forward contract for Story 2-3.

## Why This Story Exists

**G3 is the keygen byte-identity gate.** Downstream Stories 2-3 (signer) and 2-4 (pk-transform + integration) require byte-identical keys to ETHFALCON's `.rsp` vectors so their G4/G5 assertions are not poisoned by upstream drift.

**Two verified facts (see A-005 §Evidence for the full proof chain):**

1. **`noble.falcon512.keygen(innerSeed)` is byte-identical to ETHFALCON's keygen** under the same 48 B `inner_seed`. Verified empirically against vec-0's KAT.
2. **`rngAesCtrDrbg256(seed)` (from `@noble/ciphers/aes.js`) is byte-identical to ETHFALCON's Python `AES256_CTR_DRBG(seed)`.** Same NIST SP 800-90A algorithm, same AES-256-ECB primitive. Vec-0 empirical test (3 draws = 48 + 40 + 48 B) byte-matched in full.

**Consequence:** Story 2-1 is a thin TS wrapper — two ~15 LOC files + a KAT test that derives `innerSeed` at test time from the existing `drbgSeed` fixture field via `rngAesCtrDrbg256`. Story 1-2's G1 Keccak-PRG module is **NOT consumed** by this story — Keccak-PRG is the signing XOF, not the keygen XOF. G1 remains load-bearing for Stories 2-3 and 2-4.

## User Story

As a signer-catalogue maintainer, I want `keygenInternal(innerSeed)` to wrap noble's Falcon-512 keygen, with the DRBG-derived inner seed computed on the TS side at test time via `rngAesCtrDrbg256`, so G3 passes byte-identity over ≥100 `.rsp` vectors without adding any cryptographic primitive OR fixture schema field to the project.

### TS wrapper contracts (T2)

```ts
// test/signers/falcon-eth.kat-internal.ts (new, ~15 LOC)
import { falcon512 } from "@noble/post-quantum/falcon.js";
import { SignerInputError } from "./errors.js";  // or wherever the project's error type lives

export interface FalconEthKeypair {
  readonly publicKey: Uint8Array;   // 897 B
  readonly secretKey: Uint8Array;   // 1281 B
}

export function keygenInternal(innerSeed: Uint8Array): FalconEthKeypair {
  if (!(innerSeed instanceof Uint8Array) || innerSeed.length !== 48) {
    throw new SignerInputError(
      "INVALID_INNER_SEED_LENGTH",
      `expected 48 B Uint8Array, got ${innerSeed instanceof Uint8Array ? `${innerSeed.length} B` : `non-Uint8Array (${typeof innerSeed})`}`
    );
  }
  return falcon512.keygen(innerSeed);
}
```

```ts
// test/signers/falcon-eth.ts (new, ~15 LOC)
import { falcon512 } from "@noble/post-quantum/falcon.js";

export interface FalconEthKeypair {
  readonly publicKey: Uint8Array;
  readonly secretKey: Uint8Array;
}

export function keygen(): FalconEthKeypair {
  const seed = new Uint8Array(48);
  globalThis.crypto.getRandomValues(seed);
  return falcon512.keygen(seed);
}
```

> **Note on `FalconEthKeypair` / `SignerInputError`:** if the project already has a shared `Keypair` type or `SignerInputError` class in `test/signers/`, import from there instead of re-declaring. Check `ml-dsa-eth.ts` / `ml-dsa-eth.kat-internal.ts` for precedent during T2. Do NOT create a new shared module just to host these.

### TS-side DRBG derivation at test time (T3)

```ts
// Inside test/signers/falcon-eth.keygen.kat.test.ts
import { rngAesCtrDrbg256 } from "@noble/ciphers/aes.js";
import { hexToBytes } from "viem";

// Per vector (inside the test loop):
const drbg = rngAesCtrDrbg256(hexToBytes(v.drbgSeed));
const innerSeed = drbg.randomBytes(48);
const { publicKey, secretKey } = keygenInternal(innerSeed);
// assert byte-identity vs hexToBytes(v.publicKey) and hexToBytes(v.secretKey)
```

**Why this is correct:** `rngAesCtrDrbg256` is byte-identical to ETHFALCON's Python `AES256_CTR_DRBG` (vec-0 verified 2026-04-20, see A-005 Evidence §5). Same NIST SP 800-90A algorithm, same AES-256-ECB primitive. The `innerSeed` computed TS-side equals the Python-computed one byte-for-byte; `noble.falcon512.keygen(innerSeed)` then produces `{publicKey, secretKey}` matching `.rsp` fields byte-for-byte.

### Noble version pinning

Any version bump that changes `falcon512.keygen` encoder output, NTRU Gaussian sampler semantics, or `rngAesCtrDrbg256` algorithm will break G3. Treat the pin as a Gate-5 dependency; flag a version bump as a Rule-3 amendment. For `rngAesCtrDrbg256` specifically: it's a first-class public export (not `__tests`), so API-stability risk is standard semver (minor bumps should not break behavior).
