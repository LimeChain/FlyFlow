# Story 1-2: `keccak-prg-verification`

## Why This Story Exists

**G1 is the non-skippable primitive gate for the entire falcon-eth oracle chain.** DD-13 LOCKED says so, and it is recorded in the architecture document's Testing Strategy row for G1 as well as Component Decomposition row "Keccak-CTR-PRNG primitive" (line 29): before any downstream gate (G3 keygen, G4 signer, G5 pk-transform) can be trusted to produce byte-identical output against ETHFALCON's Python reference, the XOF primitive those ports consume must itself be proven byte-identical to ETHFALCON's wrapper. The shared `keccakXofFactory` at `test/signers/mldsa-encoding.ts:92` — and its underlying `createKeccakPrg` primitive at `test/signers/keccak-prg.ts` — were ported during mldsa-eth against ETHDILITHIUM's `Keccak256PRNG(a, b)` wrapper. ETHDILITHIUM's wrapper was itself copied from ETHFALCON's `keccak_prng.py` (ZKNox lineage), so internal mechanics are almost certainly identical — but the **wrapper APIs differ** (ETHDILITHIUM has a 2-arg `Keccak256PRNG(a, b)` wrapper class, ETHFALCON has a 0-arg `KeccakPRNG()` class). That delta alone is enough reason to witness byte-identity against the ETHFALCON variant explicitly, so the downstream gates can't fail spuriously on a silent wrapper-level divergence.

A failure here invalidates Story 2-1 (G3 keygen over 100 vectors) and Story 2-3 (G4 signer over 100 vectors) — both consume `keccakXofFactory` transitively through `falcon-eth.core.ts`. If G1 diverges, DD-13 says falcon-eth must port a `falconKeccakXofFactory` adapter (scope-expansion of this story) before Stories 2-1 and 2-3 can proceed. If G1 is identical (the likely outcome), this story ships a thin assertion test + a fresh fixture corpus captured from ETHFALCON and clears the path.

This story does NOT port a new PRG. It does NOT touch any signer module. It does NOT run G2/G3/G4/G5/G6. It adds (a) a new G1 vectors fixture captured from ETHFALCON's `KeccakPRNG()` Python class, (b) a loader helper for that fixture, and (c) one new test file `test/signers/keccak-prg.falcon.kat.test.ts` (name dictated by architecture §"Testing Strategy" row "G1 — Keccak-PRG vs ETHFALCON") that drives `createKeccakPrg` through the scripted inject/flip/extract sequences and asserts byte-identity against the Python-captured outputs using `assertBytesEqual` with the `"keccak-prg"` `xofId` — producing the `(factory=keccak-prg)` discriminant on divergence.

## User Story

As a signer-catalogue maintainer, I want confirmation that the shared `keccakXofFactory` produces byte-identical output to ETHFALCON's `Keccak256PRNG(a, b)` Python wrapper, so that the XOF primitive is trusted for keygen, HashToPoint, signer, and pk-transform gates.

The user story is preserved verbatim-from-plan above; see `docs/amendments.md` §A-004 for the authoritative class-name correction. T1/T2 implementation uses the correct `KeccakPRNG()` class.

### A-005 lesson generalized — "spec mirror" bug class

From `docs/amendments.md` §A-002 (drafted this feature) and the retained A-005 precedent: any port that is authored against a single upstream spec document rather than the executable source can develop a "spec mirror" bug where both TS and Python look correct in isolation but silently disagree at a layer the spec doesn't enumerate (A-002's case was the `.rsp` `sm` layout; A-005's was DRBG state advancement). The PRG-level equivalent would be, for example: an `extract()` call sequence where the out-buffer-pos persistence looks correct in spec-derived tests but diverges from the executable Python when chunked with specific block boundaries. G1's role is exactly to force the TS port against live Python output, not against a spec-derived assumption.

**Mitigation reinforced by this story:** capture is driven by running ETHFALCON's `keccak_prng.py::KeccakPRNG` directly via `python3 -c` subprocess (the existing CLI pattern). No hex literals are typed into a file by a human; every expected output is computed by running the reference code that ships in the pinned `ETHFALCON/pythonref/` submodule (SHA `03ed0d60c67087527de7c4a3c1c469b89611bd68`).

### Under-test PRG surface (inlined from `test/signers/keccak-prg.ts`)

The implementer drives this module and asserts against Python output; no modifications to it. Exported surface (verified at story creation):

```ts
export type PrgLifecycleCode =
  | "PRG_INJECT_AFTER_FLIP"
  | "PRG_EXTRACT_BEFORE_FLIP"
  | "PRG_DOUBLE_FLIP"
  | "PRG_BUFFER_OVERFLOW";
export class PrgLifecycleError extends Error { readonly code: PrgLifecycleCode; /* ... */ }
export interface KeccakPrg {
  inject(data: Uint8Array): void;
  flip(): void;
  extract(length: number): Uint8Array;
  update(data: Uint8Array): void;              // SHAKE-parity alias for inject
  read(length: number): Uint8Array;            // SHAKE-parity alias for extract
}
export function createKeccakPrg(seed?: Uint8Array): KeccakPrg;
```

Construction semantics preserved from ETHFALCON's `keccak_prng.py`:
- `createKeccakPrg(seed)` is equivalent to `const p = createKeccakPrg(); p.inject(seed);` — **does NOT auto-flip**.
- Internal state: `buffer: Uint8Array(4096)`, `bufferLen`, `state: Uint8Array(32)`, `counter: bigint` (extract-phase), out-buffer pos/len for cross-extract persistence.
- `flip()` computes `state = keccak256(buffer[:bufferLen], "bytes")`. One-shot.
- `extract(n)` drains existing out-buffer slack first (so `extract(5) + extract(27)` ≡ `extract(32)` on a fresh instance), then loops `outBuffer = keccak256(state ‖ u64_be(counter), "bytes")` with `counter` starting at 0 and incremented per block.
- Lifecycle errors throw `PrgLifecycleError` with the codes above. G1 test does NOT exercise these (they were covered by mldsa-eth Story 2's lifecycle unit tests on the same primitive); G1 exercises only the byte-identity surface.

### ETHFALCON Python reference — authoritative source (VERIFIED at story creation)

The class the G1 test asserts against. From `ETHFALCON/pythonref/keccak_prng.py` at pinned submodule SHA `03ed0d60c67087527de7c4a3c1c469b89611bd68`:

```python
# ETHFALCON/pythonref/keccak_prng.py:13-29
class KeccakPRNG:
    def __init__(self):                        # Line 14 — NO args. See Risks §1.
        """ Initialize a Keccak PRNG context. """
        self.buffer = bytearray(MAX_BUFFER_SIZE)   # MAX_BUFFER_SIZE = 4096
        self.state = bytearray(KECCAK_OUTPUT)      # KECCAK_OUTPUT = 32
        self.buffer_len = 0
        self.counter = 0
        self.finalized = False
        # Output buffer management
        self.out_buffer = bytearray(KECCAK_OUTPUT)
        self.out_buffer_pos = 0
        self.out_buffer_len = 0

    @classmethod
    def new(self):                              # Line 27 — alternate ctor, alias for KeccakPRNG()
        return self()

    def inject(self, data: bytes): ...          # Line 31 — absorb; ValueError after finalize
    def flip(self): ...                         # Line 42 — state = keccak256(buffer[:buffer_len])
    def extract(self, length: int) -> bytes: ... # Line 58 — streaming extract
    def update(self, data: bytes): ...          # Line 110 — SHAKE-parity alias (calls inject)
    def read(self, length: int) -> bytes: ...   # Line 114 — SHAKE-parity alias (calls extract)
```

**Algorithm the TS port must byte-match** (verified directly from the Python source, NOT a spec document):
1. `inject(data)`: `buffer[buffer_len : buffer_len+len(data)] = data; buffer_len += len(data)`. Error if `finalized` or cumulative length would exceed `MAX_BUFFER_SIZE=4096`.
2. `flip()`: `self.state = keccak.new(digest_bytes=32).update(buffer[:buffer_len]).digest()`. One-shot; error if already finalized.
3. `extract(length)`: drain `out_buffer[out_buffer_pos : out_buffer_len]` first; then loop `block = state ‖ struct.pack(">Q", counter)` (8-byte BIG-endian u64), `out_buffer = keccak256(block)`, copy min(remaining, 32) bytes, advance `out_buffer_pos`, set `out_buffer_len=32`, increment `counter`.

**Counter endianness and start-value are load-bearing** — `struct.pack(">Q", counter)` is big-endian, counter starts at 0, increments after each block.
