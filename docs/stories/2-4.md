## Architecture Guardrails

### G5 pk-transform — two exports in `falcon-eth.core.ts`

`test/signers/falcon-eth.core.ts` grows TWO new exports (appended to the existing file — Story 2-2 landed `hashToPointEVM`, Story 2-3 T2 landed `falcon512paddedEth`). Both read the 897 B raw Falcon-512 NIST public key, apply forward NTT, compact 16 coefficients × 14 bits (or 16 bits if the existing helper pads) into 32 `uint256` words, and emit in two shapes:

```ts
// Internal helper — returns the bigint[] of length 32, one bigint per uint256 word.
// G5 structural sub-check (AC-2) asserts length === 32 AND every element < 2^256.
// Debug anchor when AC-1 Hex comparison fails: the intermediate shape localizes
// coefficient-order / NTT-direction / bit-packing bugs before the abi.encode layer.
export function pkToNttCompact(
  rawPk: Uint8Array,
  xofFactory: XofFactory,  // keccakXofFactory for falcon-eth — shared with mldsa-eth path
): bigint[];

// Public surface — composes `pkToNttCompact` + `encodeAbiParameters([{type:"uint256[]"}], [...])`
// Returns `viem Hex` — directly passable to `verifier.setKey(hex)`.
// MIRRORS mldsa-eth's `preparePublicKeyForDeployment` SHAPE (NFR-11 cross-scheme symmetry):
//   - Same function name
//   - Same `Hex` return type
//   - Same caller pattern: `const payload = preparePublicKeyForDeployment(rawPk, factory); await verifier.write.setKey([payload])`
// AC-1 (G5 KAT) — see amendment A-007: compares structurally via `decodeAbiParameters`
// on BOTH sides (output uses dynamic `uint256[]`; fixture `reshapedPublicKey` uses fixed
// `uint256[32]`; both wrap the same 32 `uint256` coefficients). Element-wise equality
// on the decoded `bigint[]`s is the semantic oracle. `docs/amendments.md` §A-007.
export function preparePublicKeyForDeployment(
  rawPk: Uint8Array,
  xofFactory: XofFactory,
): Hex;
```

**Reuse of existing `encodePublicKeyForZKNOX` (Story 1-1 / `falcon-encoding.ts:128`):** that helper ALREADY implements the raw→NTT→compact→abi.encode transform for the non-ETH `falcon` path. Reuse it rather than re-port — the byte-difference between falcon and falcon-eth at the pk-transform layer is ZERO (both schemes use the same `compactPoly256` packing over the same NTT-domain `h`). `preparePublicKeyForDeployment` can delegate: wrap `encodePublicKeyForZKNOX(rawPk)` with an `xofFactory` signature-compatible shim OR pass `xofFactory` through unused. The `xofFactory` parameter exists for NFR-11 shape symmetry with ml-dsa-eth's binding (which DOES use the factory internally for XOF-driven NTT ingestion of `aHat`) — falcon's NTT has no XOF dependency at pk-transform time (it's a deterministic forward-NTT of coefficients parsed from the public key). Two defensible designs:
1. **Delegate-and-ignore:** `preparePublicKeyForDeployment(rawPk, _xof) = encodePublicKeyForZKNOX(rawPk)`. Simplest. Single source of truth at `falcon-encoding.ts:128`.
2. **Delegate-and-pass-through:** introduce the `xofFactory` param on `pkToNttCompact` and thread through a null/no-op implementation. Preserves the symmetry for a future NFR-11 structural grep.

Implementer decides at T1 based on whether the bench harness / fixture factory already satisfies the NFR-11 shape grep. Default to (1) unless code review demands (2).

### G6 `FalconEthAccount` — structurally identical to `MlDsaEthAccount`

Contract at `contracts/FalconEthAccount.sol`. The Solidity shape is fully specified in `docs/architecture.md` §"Smart Contract Interfaces" (lines 269-297) — inline the binding pieces here:

```solidity
// SPDX-License-Identifier: GPL-3.0
pragma solidity 0.8.34;

import {SimpleAccount} from "@account-abstraction/contracts/samples/SimpleAccount.sol";
import {IEntryPoint} from "@account-abstraction/contracts/interfaces/IEntryPoint.sol";
import {PackedUserOperation} from "@account-abstraction/contracts/interfaces/PackedUserOperation.sol";
import {SIG_VALIDATION_SUCCESS, SIG_VALIDATION_FAILED} from "@account-abstraction/contracts/core/Helpers.sol";
import {ZKNOX_ethfalcon} from "./imports/FalconRef.sol";  // NOT direct ETHFALCON import; use wrapper

/// @title FalconEthAccount
/// @author pqc-4337-laim
/// @notice ERC-4337 v0.7 account delegating Falcon-ETH signature verification
///         to the ZKNoxHQ `ZKNOX_ethfalcon` verifier (Keccak-PRG HashToPoint
///         variant). Stores a 20-byte SSTORE2 pointer (publicKeyPointer)
///         returned by `falconEthVerifier.setKey(preparedPk)`.
/// @custom:experimental This library is not audited yet, do not use in production.
contract FalconEthAccount is SimpleAccount {
    error SignatureMalformed();
    ZKNOX_ethfalcon public immutable falconEthVerifier;  // DD-9 per-account verifier
    bytes public publicKeyPointer;                        // 20-byte SSTORE2 pointer
    bytes4 private constant _VERIFY_SELECTOR =
        bytes4(keccak256("verify(bytes,bytes32,bytes)"));

    constructor(IEntryPoint anEntryPoint, ZKNOX_ethfalcon _verifier)
        SimpleAccount(anEntryPoint)
    {
        falconEthVerifier = _verifier;
    }

    /// @dev `address owner_` parameter unused by Falcon path (identity = publicKeyPointer).
    ///      Do NOT forward to `super.initialize(...)` — that would create a misleading
    ///      dual-identity model. Shadow-discipline matches MlDsaEthAccount.
    function initialize(address, bytes calldata _publicKeyPointer)
        public initializer
    {
        publicKeyPointer = _publicKeyPointer;
    }

    function _validateSignature(
        PackedUserOperation calldata userOp,
        bytes32 userOpHash
    ) internal view override returns (uint256) {
        try falconEthVerifier.verify(publicKeyPointer, userOpHash, userOp.signature)
            returns (bytes4 result)
        {
            return result == _VERIFY_SELECTOR
                ? SIG_VALIDATION_SUCCESS
                : SIG_VALIDATION_FAILED;
        } catch {
            revert SignatureMalformed();
        }
    }
}
```

**Key invariants (AC-mapped):**
- **State variable naming:** `falconEthVerifier` (mirrors `dilithiumEthVerifier` in `MlDsaEthAccount`). NOT `verifier` (too generic). NOT `falcon_eth_verifier` (snake_case — AC-10 grep fails).
- **`@custom:experimental` NatSpec tag:** required on the contract header (mirrors architecture.md line 273 + ml-dsa-eth precedent). AC-12 README runbook mentions this same tag.
- **`_VERIFY_SELECTOR` pattern:** computed via `keccak256("verify(bytes,bytes32,bytes)")` because `ZKNOX_ethfalcon.verify.selector` is ambiguous under ADL — the verifier exposes multiple `verify` overloads (`MlDsaEthAccount:37-38` established this pattern for ETH-variants).
- **`try/catch` → `SignatureMalformed()`:** the malformed-signature path relies on the verifier reverting internally (e.g., calldata length mismatch triggers BytesLib OOB). Account contract's `catch` converts the revert into a typed custom error. AC-6's dual-path walker binds to this.
- **Wrapper path, not direct submodule import:** the contract imports `ZKNOX_ethfalcon` from `contracts/imports/FalconRef.sol` — which already re-exports the submodule type (line 29 `contract ZKNOX_ethfalcon is _ZKNOX_ethfalcon {}`). AC-14 asserts no new RefShim file is created in this story; `FalconRef.sol` stays byte-identical at lines 1-29 (A-NFR-10 MIT header preservation).

### G6 dual-path walker — AC-6 malformed-signature binding

The malformed-signature rejection path uses the exact walker pattern established at `test/accounts/mldsa-eth-failures.test.ts:279-310` (see reference snippet at `mldsa-eth-failures.test.ts:244-313`). The walker MUST bind to `accountAddress.toLowerCase()` because THREE contracts now declare `SignatureMalformed()` (MlDsaAccount, FalconAccount, MlDsaEthAccount) — FOUR after this story (FalconEthAccount). Same 4-byte selector `0x2c3c2fe1` across all four. Without the address bind, a test-setup mistake routing to the wrong account would satisfy the predicate spuriously.

Inline pattern (verbatim from ml-dsa-eth-failures precedent, re-anchored for falcon-eth):

```ts
const accountAddress = account.address.toLowerCase();

await assert.rejects(
  () => simulateValidateUserOp(account, entryPoint.address, malformed, userOpHash),
  (err: unknown) => {
    if (!(err instanceof BaseError)) throw err;
    const message = err.message.toLowerCase();
    const boundToAccount = message.includes(accountAddress);

    // Canonical viem path — decoded ContractFunctionRevertedError.
    const revert = err.walk(
      (e) => e instanceof ContractFunctionRevertedError,
    ) as ContractFunctionRevertedError | null;
    if (revert?.data?.errorName === "SignatureMalformed" && boundToAccount) {
      return true;
    }

    // HH3 EDR fallback — message-regex (viem doesn't populate errorName here).
    return (
      /custom error 'signaturemalformed\(\)'/.test(message) && boundToAccount
    );
  },
);
```

> Ref: `test/accounts/mldsa-eth-failures.test.ts:244-313` — binding walker precedent. Story 2-4's T5 imports + re-anchors for FalconEthAccount. Do NOT redefine the walker inline without the address bind.

### Retrospect candidate

Proposal: universal rule **"Capstone integration stories — treat dual-gate surface as a single atomic delivery."** Story 2-4's scope crosses TS pk-transform (G5), Solidity account (G6), TS fixture factory, TS dispatcher, bench harness, report renderer, README, and naming grep. The temptation is to split into 2-4a + 2-4b, but the dispatcher extension (T6) depends on all T1-T5 falcon-eth surfaces, and the README runbook (T7) depends on refreshed bench numbers from T6. Splitting adds a wave + creates an intermediate commit where the dispatcher is extended but the bench can't prove it works end-to-end.

The size-vs-atomicity tradeoff is real: L-sized stories have more surface for mid-story scope creep (e.g., discovering AC-6 needs a new walker utility → adding to T5 vs deferring). Recommendation: at story creation, enumerate ALL AC ↔ task mappings + structural checks (file SHA regression) BEFORE T1 starts; if the enumeration exceeds the budget of what a single implementer can hold simultaneously, that's the split signal — otherwise keep the dual-gate surface atomic.

Lands at Gate 5 `[L] Apply fixes + learn` if code-review surfaces the knowledge gap (expected: learning about bench determinism gotchas, calldata-gas distribution in NTT-compact vs Algorithm-17 compressed, or walker-reuse patterns).

### Retrospect candidate — late-wave test-count regressions

Capstone stories land large test-count deltas (Story 2-4 adds ~215 tests across 5 files). A single mis-import or mis-AC-count can silently under-deliver. Recommendation: at Gate 5, compare pre-story test count (from `docs/state.json` `metrics.currentStory.testCount` or equivalent) against post-story expected delta (G5 KAT count + G6 happy count + G6 failure count + AC-FLOW-1 + README + registry asserts).
