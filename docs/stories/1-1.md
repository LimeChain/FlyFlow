# Story 1-1: `falcon-eth-fixtures`

## Why This Story Exists

This is the Wave 1 foundation for the falcon-eth feature. Every downstream gate (G1 PRG byte-identity, G2 HashToPoint byte-identity, G3 keygen, G4 signer, G5 pk-transform, G6 on-chain) depends on (a) a ground-truth KAT corpus with pinned submodule provenance and (b) a scheme-typed loader. Without this story, no downstream gate has anything authoritative to assert against.

The story ships two fixture files, a 10-LOC Solidity extension, a pure rename that separates concerns for git-bisect, and a multi-submodule-aware loader with a TypeScript discriminated overload. It lands five task-granular commits per NFR-12.

It does NOT run any G1–G6 oracle gate, introduce any new TS signer code, or touch `FalconEthAccount.sol`. Those are downstream (Stories 1-2, 2-1, 2-2, 2-3, 2-4).

## User Story

As a signer-catalogue maintainer, I want the falcon-eth KAT corpus captured and loadable through a scheme-typed API, so that every downstream oracle gate has a ground-truth dataset and a type-safe loader.

### A-005 lesson — PRE_G4_DRBG_PROBE (load-bearing)

Mldsa-eth's A-005 amendment discovered that `AES256_CTR_DRBG.random_bytes(N)` invokes `__ctr_drbg_update` at the end of every call — so naive slicing of a single `random_bytes(64)` ≠ two sequential `random_bytes(32)` calls. Falcon-ETH's `.rsp` is driven by the same DRBG. Before bulk fixture generation, a probe on vec 0 must re-derive `(sk, msg, sig)` using the exact Python `random_bytes(N_i)` call sequence from ETHFALCON's `falcon.py`. Probe failure HALTs — do not proceed to Task 1's bulk write with a wrong entropy-order assumption.

**AMENDED by `docs/amendments.md` §A-002 (2026-04-19, Rule 3):** ETHFALCON `.rsp` `sm` layout is `slen(2) || salt(40) || msg(mlen) || header(1) || esig`, NOT NIST `sig || msg`. The original "re-computed signature byte-equals `sm[:-mlen]`" predicate is superseded by a three-window equality check replicating `ETHFALCON/pythonref/test_falcon_KAT.py::test_KAT_ETH`:
1. `recoveredPk.pk == PublicKey.from_bytes(expected_pk).pk` (keygen NTRU gen determinism);
2. `py_sig[1:41] == sm[2:42]` (salt derivation — the first `random_bytes(SALT_LEN)` call during sign);
3. `py_sig[41:] == esig[1:1+len(py_enc_s)]` where `esig = sm[42+mlen : 42+mlen+sig_len]`, `sig_len = (sm[0]<<8)|sm[1]` (Gaussian body — subsequent `random_bytes` calls).

Probe implementation: one-shot Python subprocess (via `python3 -c`), imports `AES256_CTR_DRBG`, `SecretKey`, `PublicKey`, `ntru_gen`, `SHAKE`, `KeccakPRNG` from `ETHFALCON/pythonref/`, runs the ETH-variant sign flow (`xof=KeccakPRNG`) with the DRBG seeded from the `.rsp`'s `seed` field, emits `{recoveredPk: hex, stockSig: hex, salt: hex, enc_s: hex}` via stdout JSON. TS side parses `sm` per the ETHFALCON layout and asserts the three windows above.

### NFR-9 LOCKED — Runtime-gated test-override safety

Any test-only env-var hooks in the fixture-gen CLI (e.g., override SHA validation, override DRBG seed, mock Python invocation) MUST be (a) regex-validated at ingest with the pattern recorded in code, AND (b) gated behind an off-by-default sentinel env var such as `ALLOW_TEST_OVERRIDES=1`. ### Fixture schema — falcon-eth vectors.json

Exact shape (DD-7/DD-14 LOCKED, this phase):

```jsonc
{
  "scheme": "falcon-eth",
  "params": "falcon-512-keccak",
  "submoduleSource": "ethfalcon",
  "submoduleSha": "<40-hex>",
  "generatedAt": "<ISO 8601>",
  "source": {
    "rspFile": "ETHFALCON/test/ethfalcon512-KAT.rsp",
    "drbgDerivation": "AES256_CTR_DRBG(drbgSeed).random_bytes(N) — N determined by PRE_G4_DRBG_PROBE",
    "ctx": "0x"
  },
  "vectors": [
    {
      "id":                "vec-000",
      "drbgSeed":          "0x…(48B)",
      "publicKey":         "0x…(897B raw)",
      "secretKey":         "0x…",
      "reshapedPublicKey": "0x…(abi.encode(uint256[32]))",
      "message":           "0x…",
      "signature":         "0x…(1064B = salt(40)‖s2_compact(1024))"
    }
  ]
}
```

Key invariants:
- Hex-encoded `0x`-prefixed lowercase (viem convention).
- `submoduleSource: "ethfalcon"` is mandatory — it's the discriminator the multi-submodule loader uses (AC-4).
- `signature` is raw concat, NOT abi-encoded. Salt is embedded at offset 0..40 inside the 1064-byte signature — NOT a separate top-level field (the `.rsp`'s `sm` packs it inside).
- `reshapedPublicKey` is pre-`abi.encode`'d — G5 downstream asserts this byte layout directly.

## Risks / Gotchas

- **`git submodule status` leading prefix:** output is of the form `" <sha> <name> (heads/main)"` with a leading status character: `+` (HEAD differs), `-` (not initialized), `U` (merge conflicts), or space (clean). The multi-submodule loader's SHA-equality comparison MUST strip this prefix before comparing — the existing ETHDILITHIUM-only loader at `test/fixtures/kat/index.ts:190` handles this via `.replace(/^[+\-U ]/, "")`; the Task 4 refactor must preserve this strip on BOTH branches. Missing the strip causes a silent always-mismatch when the submodule is in `+` state (common during rebases).
- **Canonical serialization drift:** Task 4's backfill of `submoduleSource: "ethdilithium"` into `test/fixtures/kat/mldsa-eth/vectors.json` must preserve canonical key ordering, indentation, and hex case. A byte-noisy diff breaks future bisect. Re-run the CLI's serializer rather than hand-editing.
- **Commit granularity (DD-26 + NFR-12):** Task 3's commit is the weird-looking one — 8 files touched, zero net behavior change. Reviewers may push to squash it into T4. DO NOT. The whole point is a surgical bisect surface. Message body should explicitly cite DD-26.
- **PRE_G4_DRBG_PROBE failure path:** if the probe fails on vec 0, do NOT ship a best-effort `vectors.json` with bogus `reshapedPk` fields. HALT, log to `docs/amendments.md` with an A-005-equivalent entry, and escalate. Shipping partial/incorrect fixtures silently poisons all downstream gates.
- **ZKNOX_HashToPointExposed is `pure` — but `.compute()` STILL executes on-chain:** the fixture generator must deploy the contract to a Hardhat node and call `.compute()` as a `staticCall` (or a read-only `call` via viem) — no transaction, zero gas cost in measurement, but the underlying keccak256 loop still runs. Don't assume `pure` means "compile-time constant-folded."
- **AC-7 bisect surface integrity:** if the implementer discovers a rename miss in Task 3 AFTER Task 4 is committed, the fix-up goes into a NEW separate commit (NOT amended into either T3 or T4). DD-26's bisect-surgery property holds only as long as no commit is retroactively rewritten.
