# Story 2-2: `hashtopoint-port`

## Why This Story Exists

**G2 is the HashToPoint byte-identity gate for the falcon-eth oracle chain.** DD-25 LOCKED fixes the trust anchor as the **pinned Solidity `hashToPointEVM` free function at `ETHFALCON/src/ZKNOX_HashToPoint.sol:22`** (Option C, fresh-generation), NOT a Python spec document and NOT a hand-transcribed corpus. Story 1-1 Task 2 landed the 8-vector fixture at `test/fixtures/kat/falcon-eth/hashtopoint-vectors.json` by deploying `ZKNOX_HashToPointExposed` in Hardhat and capturing `.compute(salt, msg)`; Story 1-1 Task 4 landed the loader `loadHashToPointVectors(): HashToPointVector[]` at `test/fixtures/kat/index.ts:607`. Story 1-2 closed G1, so the shared Keccak primitive (`keccak-prg.ts`, which wraps `viem`'s `keccak256`) is proven byte-identical to ETHFALCON's `KeccakPRNG()` Python wrapper.

This story's purpose is narrow: port the Solidity `hashToPointEVM` algorithm to TypeScript as a standalone shared-core export at the **new file** `test/signers/falcon-eth.core.ts`, and prove byte-identity over the 8 vectors via a new KAT test. **G2 must pass BEFORE Story 2-3 begins** — Story 2-3's signer state machine consumes `hashToPointEVM` at the `c = HashToPoint(r‖m, q, n)` step, and a byte-wrong port there would silently poison every G4 assertion (and the debug effort would be forced up into the signer layer, which is ~20× more surface area to bisect). A tight ~20-LOC isolated G2 test pays for itself the first time a chunk-endianness or rejection-threshold bug is caught here instead of at G4.

Because G1 already witnessed byte-identity for the shared Keccak primitive, **any G2 divergence is an unambiguous port bug** in this story's output — not a primitive bug. AC-2's error-path hints (chunk endianness, 61445 threshold, mod-q reduction, absorb order, coefficient-order) enumerate exactly the bug classes the port surface opens. The story creates the `falcon-eth.core.ts` file seed that downstream Stories 2-1 (keygen), 2-3 (signer), and 2-4 (pk-transform) will each append to — this story ships only the HashToPoint export and the module-header skeleton.

### Keccak primitive authority (DD-1 LOCKED) — resolving the plan's "using `keccak-prg.ts` internally" wording

**The plan at line 115 says "using `keccak-prg.ts` internally". The architecture at line 401-404 clarifies this as "keccak256-counter-mode construction (NOT the Keccak-PRG primitive)".** These are not contradictory once the primitive distinction is drawn:

- The Solidity loop body is `buffer := keccak256(extendedState, 40)` — a **one-shot hash** of the full 40-byte `state‖counter_u64` input per iteration, NOT a streaming PRG inject/flip/extract session.
- `createKeccakPrg` at `test/signers/keccak-prg.ts:92` is a **three-phase state machine** (inject → flip → extract) that streams block-aligned output from a single absorbed buffer. Using it here would be byte-wrong: `extract()` re-uses the SAME internal `state` across blocks (only the counter advances), which exactly matches the Solidity loop IF we set `state = keccak256(salt‖msg)` upfront and then call `extract(16 * 2)` chunks — but the shape doesn't match cleanly (the Solidity loop also does rejection sampling inside each 32-byte buffer, so we'd need to round-trip through the buffer anyway).
- **The authoritative TS Keccak primitive for this port is `keccak256` from `viem`** — verified at story creation as the primitive that `keccak-prg.ts` itself imports (`test/signers/keccak-prg.ts:30` → `import { keccak256 } from "viem"`). Story 1-2 closed G1 by witnessing `keccakXofFactory` (the factory wrapping `createKeccakPrg`) byte-identical to ETHFALCON's `KeccakPRNG()` Python wrapper. Since the Python wrapper uses `pycryptodome.keccak.new(digest_bytes=32)` and the TS primitive uses `viem.keccak256` (both the Keccak-f[1600] permutation with the 0x01 padding — 256-bit digest variant), **G1's byte-identity assertion transitively covers `viem.keccak256` for this port.** No separate primitive verification needed.

**Implementer decision (binding):** use `import { keccak256 } from "viem"` for both the initial `state = keccak256(salt‖msg)` and the per-iteration `buffer = keccak256(extendedState)`. Do NOT construct a `createKeccakPrg()` instance and drive it through inject/flip/extract — that's the wrong primitive shape, AND it would be indirect (a wrapper around the same `viem.keccak256` the direct path calls). Reuse the same `viem.keccak256` via `viem.keccak256(input, "bytes")` to return `Uint8Array` (matches the `keccak-prg.ts:140,177` call-style for byte-output). Any mention in the plan or other upstream docs of "using keccak-prg.ts internally" is to be read as "using the same underlying Keccak primitive" — the one-shot hash, not the PRG state machine.

### Algorithm (inlined, authoritative — from `ETHFALCON/src/ZKNOX_HashToPoint.sol:22-54`)

Constants (inline — implementer does NOT need to read Solidity):

- `Q = 12289` (Falcon modulus; output coefficient bound)
- `KQ = 61445` (rejection threshold; `5 * Q` — NOT `61440`, off-by-5 is a common silent bug)
- `N = 512` (output coefficient count)

Procedure:

```
function hashToPointEVM(salt: Uint8Array, msg: Uint8Array): Uint16Array:
  // --- Initial state ---
  const initialState = keccak256(concat(salt, msg), "bytes")          // 32 bytes
  const extendedState = new Uint8Array(40)                             // 32 state + 8 counter
  extendedState.set(initialState, 0)                                   // bytes 0..32
  // counter starts at 0 at bytes 32..40 (big-endian uint64)
  // DataView.setBigUint64(32, 0n, /* littleEndian */ false) — already zero

  const output = new Uint16Array(N)                                    // 512 entries
  let i = 0
  let counter = 0n                                                     // bigint for u64 arithmetic

  // --- Main loop ---
  while (i < N):
    const buffer = keccak256(extendedState, "bytes")                   // 32 bytes, one-shot hash
    // Extract 16 chunks of 2 bytes each, BIG-ENDIAN (high bits first — see §Risks §1)
    for (let chunkIdx = 0; chunkIdx < 16; chunkIdx++):
      const byteOffset = chunkIdx * 2
      const chunk = (buffer[byteOffset] << 8) | buffer[byteOffset + 1]   // uint16 BE
      if (chunk < KQ):                                                   // rejection sample
        output[i] = chunk % Q                                            // mod-q reduction AFTER the check
        i++
        if (i === N) break                                               // done
    // --- Increment counter (big-endian u64 at bytes 32..40) ---
    counter += 1n
    new DataView(extendedState.buffer, 32, 8).setBigUint64(0, counter, /* littleEndian */ false)

  return output
```

### Why `lt(j, 666)` is NOT 666 iterations (Solidity uint-underflow guard)

The Solidity loop reads:

```solidity
for { let j := 240 } lt(j, 666) { j := sub(j, 16) } {
    let chunk := and(shr(j, buffer), 0xffff)
    // ...
}
```

`j` starts at 240 and decrements by 16 each iteration: 240, 224, 208, ..., 16, 0. On the 17th decrement, `j` underflows (EVM `sub` is unsigned — `0 - 16` becomes `2^256 - 16`), and `2^256 - 16 > 666` is TRUE in unsigned comparison, so the loop condition `lt(j, 666)` becomes FALSE and the loop exits. Net effect: **exactly 16 iterations per outer loop** (j ∈ {240, 224, 208, 192, 176, 160, 144, 128, 112, 96, 80, 64, 48, 32, 16, 0}). The TS port writes this as `for (let chunkIdx = 0; chunkIdx < 16; chunkIdx++)` — simpler and equivalent. **Do NOT port `666` as a literal constant; it's a uint-overflow guard, not a semantic bound.**

### Chunk endianness (load-bearing — see §Risks §1)

`and(shr(j, buffer), 0xffff)` at j=240 extracts bytes 0..2 of the 32-byte buffer (the HIGH-order bits); at j=224, bytes 2..4; ...; at j=0, bytes 30..32. **The first accepted chunk comes from bytes[0..2] of the buffer — big-endian — and goes to `output[i++]` starting at i=0.** Equivalent TS: `(buffer[2*k] << 8) | buffer[2*k + 1]` for `k = 0, 1, ..., 15`. Off-by-reverse (writing `buffer[2*k + 1] << 8 | buffer[2*k]` for little-endian, OR iterating chunks high-to-low-index instead of low-to-high) produces a silent divergence that AC-1 catches on vec-000.

### Counter increment — big-endian uint64 at bytes 32..40

The Solidity `counter := add(counter, 6277101735386680763835789423207666416102355444464034512896)` adds `1 << 192` to a uint256 register. The `mstore` writes that register to `extendedAdress` (which points at byte 32 of `extendedState`). On a uint256, `1 << 192` occupies bytes 0..8 of the 32-byte word (MSB-aligned) — which, when stored at memory offset 32 of `extendedState`, lands at bytes 32..40 of the concrete buffer. Net effect: the u64 at `extendedState[32..40]` is incremented by 1 per iteration, big-endian. **TS port:** `new DataView(extendedState.buffer, 32, 8).setBigUint64(0, counter, /* littleEndian */ false)` after incrementing `counter: bigint += 1n`. This mirrors `keccak-prg.ts:176` exactly (`blockView.setBigUint64(KECCAK_OUTPUT, counter, false)`).

Verification: `6277101735386680763835789423207666416102355444464034512896 === (1n << 192n)` — confirmed at story creation via `python3 -c "print(6277101735386680763835789423207666416102355444464034512896 == (1<<192))"` → `True`.

## Risks / Gotchas

1. **Chunk endianness is BIG-ENDIAN, high bits first.** `and(shr(j, buffer), 0xffff)` at `j=240` reads the TOP 16 bits of the 32-byte buffer as a uint256 — which is bytes `[0, 1]` in memory (MSB-aligned). The TS equivalent is `(buffer[2*k] << 8) | buffer[2*k + 1]` for `k = 0..15`. A little-endian port (`buffer[2*k] | (buffer[2*k + 1] << 8)`) produces silently-wrong coefficients on every vector; AC-1 catches it on vec-000 but the debug time is non-trivial if you don't know which primitive to suspect. AC-2's hint list names this first.

2. **Rejection threshold is 61445, NOT 61440.** `KQ = 5 * Q = 5 * 12289 = 61445`. Using `61440` produces a silent off-by-5: any keccak-buffer chunk in the range `[61440, 61445)` (a ~5/65536 probability per chunk, or ~0.008% of accept candidates) would get rejected under the wrong threshold AND would get accepted under the correct one — so the port's output for most vectors looks "mostly right" with occasional coefficient divergences that look like noise rather than a systematic bug. Write the constant as `const KQ = 61445` (or compute `5 * Q` inline) and grep-gate its presence via `must_haves`.

3. **Counter increment scope.** The Solidity adds `1 << 192` to a uint256 register, then stores to `extendedAdress` (byte 32 of `extendedState`). On a uint256, `1 << 192` occupies bytes `[0..8]` of the 32-byte word — which, aligned to memory offset 32, lands at bytes `[32..40]` of the concrete `extendedState` buffer. So the u64 at `extendedState[32..40]` is incremented by 1 per iteration, big-endian. **Do not port as a `Uint8Array.set` of a hand-rolled 8-byte BE encoding** — use `DataView.setBigUint64(0, counter, false)` (third arg `false = big-endian`) to avoid off-by-one/endianness bugs. This matches `keccak-prg.ts:176` verbatim. `counter: bigint` (not `number`) to avoid the JS `Number.MAX_SAFE_INTEGER = 2^53 - 1` ceiling — Falcon-512 rejection sampling usually converges in 1-2 outer iterations for random inputs, but pathological inputs could in principle loop further, and `bigint` is zero-cost for this.

4. **`keccak256(extendedState)` is a ONE-SHOT hash, not a PRG session.** Each iteration rebuilds `extendedState[32..40]` with the new counter and hashes the full 40 bytes. The port must NOT wrap `extendedState` in a `createKeccakPrg()` instance — that would drive a THREE-phase state machine (inject → flip → extract) that doesn't match the Solidity algorithm at all. DD-25 Option C: the trust anchor is the Solidity `keccak256(bytes memory)` opcode, the TS mirror is `viem.keccak256(Uint8Array, "bytes")`. The G1 byte-identity closure (Story 1-2) transitively covers this: `keccak-prg.ts` uses `viem.keccak256`, G1 proved `keccak-prg` ≡ ETHFALCON's `KeccakPRNG()`, ETHFALCON's `KeccakPRNG()` uses `pycryptodome.keccak.new(digest_bytes=32)`, the Solidity opcode is the same Keccak-f[1600] permutation — all four converge on the same underlying primitive.

5. **Coefficient order: first accepted chunk goes to `output[0]`, NOT `output[511]`.** The Solidity `offset := add(output, 32)` starts at output's first element (offset 32 skips the length prefix in Solidity's memory layout) and `mstore` + `offset := add(offset, 32)` advances forward. Coefficients are stored in acceptance order, low index to high. Off-by-reverse = silent divergence.

6. **`mod q` is applied AFTER the rejection check, on values in `[0, 61445)`.** A chunk of `12289` is `< 61445` AND equal to `q`, so `chunk % q === 0` — this produces a zero coefficient, which is intentional per the Solidity (uniformity over `Z_q` with `5 * q = 61445` as the ceiling means each residue class gets ~5 candidate chunks in `[0, 61445)`, and `0` gets exactly `{0, 12289, 24578, 36867, 49156}` as pre-images). Do NOT apply `mod q` before the rejection check — that would silently remap `61440 ≤ chunk < 61445` (which correctly fall below the threshold) into `61440 - 4*12289 = 12284 ≤ chunk < 61445 - 4*12289 = 12289` but with wrong probability mass. AC-2's hint list calls this out.

7. **Absorb order: `state = keccak256(salt ‖ msg)`, NOT `keccak256(msg ‖ salt)`.** Solidity line 28: `state = keccak256(abi.encodePacked(salt, msgHash));`. `abi.encodePacked` is order-preserving concatenation — `salt` goes FIRST in the 40+N-byte input. Swapping produces a completely different initial state, so every coefficient diverges starting at index 0 on every vector. Compare to `hashToPointTETRATION` at line 116 which intentionally swaps (`keccak256(abi.encodePacked(msgHash, salt))`) — don't mix them up.

8. **`extendedState` size is exactly 40 bytes, not 32 + bigint.** Allocate `new Uint8Array(40)`, set the initial 32-byte state via `.set(initialState, 0)`, and use the DataView to write the counter at offset 32. Don't use a `Uint8Array(32) + separate counter variable` shape — that misaligns the keccak input.

9. **Node and JS native `Uint16Array` vs `readonly number[]` comparison semantics.** The fixture `expectedHash` is `readonly number[]` (plain JS numbers, from `JSON.parse`). The port returns `Uint16Array`. `Uint16Array[k]` returns a JS `number` (auto-unboxed; 0..65535 range). `===` between two JS numbers is strict equality, works as expected.
