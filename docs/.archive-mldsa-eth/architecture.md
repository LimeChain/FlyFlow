# Architecture: ML-DSA-ETH — Keccak-Based ML-DSA for ERC-4337

## Goals & Constraints

**Drivers (from spec):** byte-compatibility with ZKNox's verifier (NFR-1), deterministic KAT byte-identity with the Python oracle (NFR-2), zero Python in shipped tree (NFR-3, C-2), tooling continuity (NFR-4, C-4, C-5), scheme extensibility without `=== 3` literals (AC-D-1), XOF isolation across NIST and ETH in one process (AC-A-1 HIGH), failure-mode parity with `MlDsaAccount`/`FalconAccount` (AC-A-2), benchmark per-scheme failure isolation (AC-A-3).

**Non-negotiable boundaries:** ZKNox submodule is external — we do not re-author or audit its Solidity. ML-DSA-44 parameters are fixed (C-6). The only XOF substitution point is Keccak-256 (C-7). Archived `pqc-4337` artifacts are frozen (C-8).

**Boundaries:**
- `mldsa-encoding.ts` is shared between `ml-dsa.ts` (NIST) and `ml-dsa-eth.ts` (Keccak). XOF is passed **as a function parameter at every call-site** — no module-level mutable state. AC-A-1 enforced by source grep for module-level `_xof` assignment.
- `MlDsaEthAccount.sol` inherits `SimpleAccount` and overrides only `_validateSignature` — identical shape to `MlDsaAccount` (DD-5).
- Python code lives exclusively under `ETHDILITHIUM/` (submodule root) — `.gitignore` already excludes it from shipped tree. NFR-3 relies on code-review discipline at PR time (user-accepted deferral).

## Data Models

Crypto types + fixture schema only — no database entities.

### `Scheme` union + `SCHEMES` const (per DD-9)
```ts
// test/signers/schemes.ts
export type Scheme = "ecdsa" | "falcon" | "mldsa" | "mldsa-eth";
export const SCHEMES = ["ecdsa", "falcon", "mldsa", "mldsa-eth"] as const
  satisfies readonly Scheme[];
```

### ETHDilithium public key (reshaped — DD-2 LOCKED)
Produced by `preparePublicKeyForDeployment(rawPk, xof)` → SSTORE2 payload:
```
abi.encode(bytes aHatEncoded, bytes32 tr, bytes t1Encoded)
```
`aHatEncoded` = 4×4 NTT-domain matrix, 32-bit compact-packed. `tr` = `Keccak256(ρ ∥ t1)`. `t1Encoded` = Power2Round (`<<D=13`) + NTT + compact-packed.

### Signature payload at Solidity boundary (DD-8 LOCKED)
```
abi.encode(bytes cTilde, bytes z, bytes h)
```
`cTilde` = 32-byte Keccak-256 challenge digest. `z` = flat z-polynomial encoding. `h` = flat hint encoding (length `k + ω` = 84). Verifier called via the 3-arg `ISigVerifier.verify(bytes pk, bytes32 m, bytes signature) returns (bytes4)` overload — matches `MlDsaAccount`.

### KAT fixture JSON (DD-7 LOCKED — sourced from ZKNox `.rsp`)
Location: `test/fixtures/kat/mldsa-eth/vectors.json`
```json
{
  "scheme": "mldsa-eth",
  "params": "dilithium2-keccak",
  "submoduleSha": "<40-hex>",
  "generatedAt": "<ISO 8601>",
  "source": {
    "rspFile": "ETHDILITHIUM/pythonref/assets/PQCsignKAT_Dilithium2_ETH.rsp",
    "drbgDerivation": "AES256_CTR_DRBG(drbgSeed).random_bytes(64) → ζ=[0:32], rnd=[32:64]",
    "ctx": "0x"  // empty — matches generate_KAT_example.py
  },
  "vectors": [
    {
      "id": "vec-001",
      "drbgSeed":          "0x…(48B)",  // audit trail only — JS never consumes this
      "zeta":              "0x…(32B)",  // G1 input  (CTR-DRBG bytes 0..32)
      "rnd":               "0x…(32B)",  // G2 input  (CTR-DRBG bytes 32..64)
      "publicKey":         "0x…",        // G1 expected + G3 input  (1312B raw Dilithium2 pk)
      "secretKey":         "0x…",        // G1 expected + G2 input  (2560B raw Dilithium2 sk)
      "reshapedPublicKey": "0x…",        // G3 expected + G4 input  (SSTORE2 payload)
      "message":           "0x…",        // G2 input
      "signature":         "0x…"         // G2 expected + G4 input  (cTilde‖z‖h concat)
    }
  ]

```
Hex-encoded bytes with `0x` prefix (viem idiom). `signature` is raw concatenated `cTilde‖z‖h`, not abi-encoded — abi-encoding is applied at the contract-call site, not in the fixture. Each vector carries data for gates G1–G4; `(ζ, rnd)` are derived once at fixture-gen time by replaying `AES256_CTR_DRBG(drbgSeed)` and consuming the first 64 bytes (matches the consumption order in `dilithium.py:399` keygen + `dilithium.py:442` sign). `drbgSeed` is retained in the fixture for independent audit — anyone can re-derive `(ζ, rnd)` by running the same DRBG. Failure-class vectors are synthesized at test time (bit-flip, wrong-key, truncate) from valid vectors — not stored. Coverage: ~100 vectors per variant inherited from the `.rsp` file (NFR-2 ≥10 minimum easily satisfied).

### PRG KAT fixture JSON (DD-11 LOCKED — new)
Location: `test/fixtures/kat/keccak-prg/vectors.json`. Schema per vector: `{id, source: "zhenfei-canonical"|"python-ref-extended", description, injects[], extracts[], expected[]}` with optional `expected_slice`/`expected_slices` for block-chain vectors. File carries `submoduleSha` (AC-U-3 enforced) and `generatedAt`.

- **Layer 1 (4 vectors, embedded hex)** — Zhenfei's canonical vectors copied verbatim from `ETHDILITHIUM/test/keccak_prng.t.sol`. Root-of-trust: ZKNox's Solidity test already verifies C ≡ Solidity against this set. Cases: single-block extract, counter-increment via extract(64), seed-length sensitivity, multi-block stream (v4 adapted to stream API via `extract(80)` + high-16 slice of each 32 B block).
- **Layer 2 (≥3 vectors, Python-generated)** — boundary cases the canonical set doesn't reach: cross-extract (`extract(5)` + `extract(27)` = `extract(32)`), multi-inject absorb concatenation, empty-seed (`keccak256(b"")` state), ML-DSA-shaped seed (`ρ ‖ j ‖ i`, extract ~408 B for ExpandA).

Layer 1 is checked in as hex literals — no Python invocation required for this subset. Layer 2 is emitted by the fixture-gen CLI at regeneration time.

> Ref: `ETHDILITHIUM/test/keccak_prng.t.sol` for the canonical vector hex bytes; `test/fixtures/kat/keccak-prg/vectors.json` is the ground-truth artifact.

## Key Workflows

**UC-1 — Sign & verify userOp (happy path).** Engineer calls `keygen("mldsa-eth")` → `{ publicKey, secretKey }`. `publicKey` passed to `preparePublicKeyForDeployment(pk, keccak256Prg)` → reshaped bytes → `MlDsaEthAccount` constructor/initialize stores pointer via SSTORE2. Engineer calls `signUserOp("mldsa-eth", sk, userOp, entryPoint, chainId)` → computes userOpHash (shared `userOpHash.ts`) → signs under Keccak-256 PRG → emits `abi.encode(cTilde,z,h)`. EntryPoint calls `account.validateUserOp` → `_validateSignature` → `verifier.verify(pk, userOpHash, sig)` → `bytes4 selector` → return `SIG_VALIDATION_SUCCESS`.

**UC-2 — Regenerate KAT fixtures from ZKNox `.rsp`.** Engineer runs `node scripts/generate-kat-fixtures.ts`. CLI (a) reads pinned submodule SHA from `.gitmodules`, (b) checks submodule HEAD matches pin (fails with regeneration guidance per AC-U-2), (c) spawns a single `python3 -c "..."` that imports `parse_kat_data`, `AES256_CTR_DRBG`, `Dilithium2`, and `Keccak256PRNG` from `ETHDILITHIUM/pythonref/dilithium_py`. For each `count` in `assets/PQCsignKAT_Dilithium2_ETH.rsp`: parse `(drbgSeed, msg, pk, sk, sm)` → `sig = sm[:-mlen]` → replay `AES256_CTR_DRBG(drbgSeed).random_bytes(64)` → `ζ, rnd = bytes[0:32], bytes[32:64]` → call `Dilithium2.pk_for_eth(pk, _xof=Keccak256PRNG, _xof2=Keccak256PRNG)` for `reshapedPk`. (d) Serializes the 8-field tuples to `test/fixtures/kat/mldsa-eth/vectors.json` with SHA embedded. Note: the `dilithium.py` consumption order is `random_bytes(32)` for ζ in keygen (line 399) followed by `random_bytes(32)` for rnd in sign (line 442) with `ctx=b""` — `(ζ, rnd) = DRBG[0:64]` is the byte-identical replay. Deterministic ordering → diffable per AC-U-4.

**UC-3 — Verify ported components against known implementations (four byte-identity gates + verifier integration).** Porting chain **PRG → keygen → signer → pk-transform → verifier**; each gate anchors the next with byte-identity-verified inputs. KAT suite loads ML-DSA fixtures via `loadKatVectors(scheme)` and PRG fixtures via `loadPrgVectors()`; both assert embedded SHA matches current submodule HEAD (AC-U-3 hard-fail). `(ζ, rnd)` are pre-derived from `drbgSeed` at fixture-gen time; JS never runs CTR-DRBG.

- **G0 (Keccak-PRG byte-identity):** fresh `KeccakPrg` per fixture, scripted `inject`/`flip`/`extract` vs expected bytes. Isolates the stream construction (Layer-1 canonical + Layer-2 boundary — see fixture schema).
- **G1 (keygen byte-identity):** `keygenInternal(zeta)` → assert `(publicKey, secretKey)` vs `.rsp` per vector. Uses G0-verified PRG. No sign, no reshape.
- **G2 (signer byte-identity):** `signWithRnd(sk, msg, rnd, ctx=0x)` → assert signature bytes vs `.rsp`. Uses G0-verified PRG + G1-verified `sk` + fixture `rnd`. No pk transform.
- **G3 (pk-transform byte-identity):** `preparePublicKeyForDeployment(publicKey, prgFactory)` → assert reshaped bytes vs `.rsp`. Uses G0-verified PRG + G1-verified `pk`. No signer.
- **G4 (verifier integration):** deploy `ZKNOX_ethdilithium.sol`, submit `(reshapedPublicKey, message, signature)` via account — assert `bytes4` match selector. Composes G0-G3; a G4 red with G0-G3 green localizes the bug to DD-8 ABI wiring, not crypto primitives.

**UC-4 — Run 4-way benchmark.** `test/bench/gas-benchmark.test.ts` iterates `SCHEMES` (from shared const). Each iteration wrapped in try/catch; failure records `{ scheme, status: "failed", reason }` and continues (AC-A-3 isolation). `generate-report.ts` reads records, emits table with 4 rows; failed rows render the reason string instead of a gas number. `results.length === SCHEMES.length` guard (no literal 3).

**UC-5 — Reject invalid signature at account boundary.** Three rejection paths:
- **Cryptographic reject** (wrong key, bit-flip over valid-length blob): verifier returns non-matching `bytes4` → account returns `SIG_VALIDATION_FAILED`.
- **Malformed** (truncated, over-length, unparseable): verifier reverts internally → account `catch`es → `revert SignatureMalformed()`.
- **Each path exercised by ≥1 test per AC-A-2** — parity with `MlDsaAccount` / `FalconAccount`.

**Suggested story decomposition (advisory — Plan phase may adjust).** Four stories, one per ported component; verifier integration folded into the last:

1. **Story 1 — Keccak-PRG port → G0 + G0-prime PASS.** Port stream construction; land Layer-1 canonical (embedded hex) + Layer-2 (Python-gen) fixtures; run G0 KAT; G0-prime deploys `ZKNOX_keccak_prng.sol` in a Hardhat test and asserts JS ≡ Solidity for Layer-2 scripts (required, not optional). ~160 LOC total (~80 PRG + ~50 KAT + ~30 Hardhat cross-check).
2. **Story 2 — XOF refactor + keygen port → G1 PASS.** `mldsa-encoding.ts` XOF-factory refactor (AC-D-2 pre-refactor 100-vector NIST regression captured from `PQCsignKAT_Dilithium2.rsp`); noble `ml_dsa44` keygen fork. ~150 LOC.
3. **Story 3 — Signer port → G2 PASS.** Rejection loop, SampleInBall, MakeHint, ExpandMask, signer encoding. ~300 LOC — largest story, rejection-loop state machine.
4. **Story 4 — pk-transform + verifier integration → G3 + G4 PASS.** `preparePublicKeyForDeployment` ETH path, `MlDsaEthAccount`, ZKNox submodule compile path, benchmark harness, failure-class tests. ~200 LOC.

The chain is strictly ordered by byte-identity dependency. PRG-as-Story-1 is load-bearing — a G0 failure localizes to one ~80 LOC module versus bisecting a ~300 LOC rejection-loop state machine at G2.

## Error Handling Strategy

### JS signer taxonomy (AC-U-1)
Extend existing `test/signers/errors.ts` pattern. Stable `code` field for programmatic matching:

| Code | Class | Condition |
|---|---|---|
| `INVALID_SECRET_KEY_LENGTH` | `SignerInputError` | `sk.length !== 2560` (Dilithium2) |
| `INVALID_MESSAGE` | `SignerInputError` | message not bytes or `0x...` hex |
| `INVALID_PUBLIC_KEY_LENGTH` | `SignerInputError` | `pk.length !== 1312` |
| `KAT_SCHEMA_MISMATCH` | `KatFixtureError` | missing required top-level keys |
| `KAT_SUBMODULE_SHA_MISMATCH` | `KatFixtureError` | embedded SHA ≠ current submodule HEAD |
| `PRG_INJECT_AFTER_FLIP` | `PrgLifecycleError` | `inject`/`update` called after `flip` |
| `PRG_EXTRACT_BEFORE_FLIP` | `PrgLifecycleError` | `extract`/`read` called before `flip` |
| `PRG_DOUBLE_FLIP` | `PrgLifecycleError` | `flip` called twice on the same instance |
| `PRG_BUFFER_OVERFLOW` | `PrgLifecycleError` | inject size would exceed `MAX_BUFFER_SIZE` |
| `INTERNAL_SIGNER_ERROR` | `SignerInternalError` | unexpected failure in XOF / NTT / encoding |

Error classes extend a `SignerError` base with `readonly code` discriminant. Tests assert on `code`, never message strings. `PrgLifecycleError` is its own class under the same base so PRG-level misuse fails loudly at the primitive instead of surfacing as a malformed-signature symptom deeper in the stack.

### Fixture-gen CLI diagnostics (AC-U-2)
Four onboarding failure modes, each emitting the exact next command:
1. Submodule uninitialized → `git submodule update --init --recursive`
2. Submodule at wrong commit → prints expected and actual SHAs
3. Python version mismatch → prints required and detected versions
4. pip deps missing → prints pinned `requirements.txt` path + install command

### Account contract reverts
`SignatureMalformed()` custom error — shared with `MlDsaAccount` / `FalconAccount` convention. No other reverts from `_validateSignature`; crypto rejection returns `SIG_VALIDATION_FAILED` (uint256 1).

### Refactor rollback (AC-D-2 pre-refactor NIST regression)
On any post-refactor byte-identity mismatch against the frozen 100-vector NIST fixture: **HALT**, bisect, or revert-and-retry on a fresh branch. AC-D-1 (existing ML-DSA suite byte-identical) is a blocking post-condition — the encoding module does not land until the 100-vector regression AND the full NIST test suite pass byte-for-byte.

## Smart Contract Interfaces

### `MlDsaEthAccount` (new)
```solidity
contract MlDsaEthAccount is SimpleAccount {
  error SignatureMalformed();
  IZKNOXEthDilithium public immutable verifier;
  bytes public publicKeyPointer;   // 20-byte SSTORE2 pointer (bytes) returned by verifier.setKey()

  function initialize(address, bytes calldata _publicKeyPointer) public initializer {
    publicKeyPointer = _publicKeyPointer; ...
  }

  function _validateSignature(PackedUserOperation calldata userOp, bytes32 userOpHash)
    internal view override returns (uint256)
  {
    try verifier.verify(publicKeyPointer, userOpHash, userOp.signature) returns (bytes4 result) {
      return result == _VERIFY_SELECTOR ? SIG_VALIDATION_SUCCESS : SIG_VALIDATION_FAILED;
    } catch { revert SignatureMalformed(); }
  }
}
```
Identical structural shape to the existing `MlDsaAccount` pattern. `_VERIFY_SELECTOR` = `ISigVerifier.verify.selector`.

### `IZKNOXEthDilithium` — verified against submodule commit
```solidity
interface IZKNOXEthDilithium {
  function verify(bytes calldata pk, bytes32 m, bytes calldata signature)
    external view returns (bytes4);
  function setKey(bytes calldata reshapedPk) external returns (bytes memory);   // returns abi.encodePacked(address)
}
```
Signatures confirmed against `ETHDILITHIUM/src/ZKNOX_ethdilithium.sol:81` (verify) and `:29` (setKey). The `bytes` return from `setKey` is `abi.encodePacked(sstore2Pointer)` — 20 bytes containing the pointer address; consumers pass it back to `verify` as the `pk` argument unchanged (assembly-extracted via `shr(96, calldataload(pk.offset))` inside the verifier). Account stores this bytes blob directly — no decode needed.

**Naming normalization (Rule 3 amendment).** The existing `MlDsaAccount.sol` and `FalconAccount.sol` store the same SSTORE2-pointer bytes under the variable name `publicKey`, which is semantically misleading (the variable holds a *pointer to* the key, not the key itself). Story 4's first task renames `publicKey` → `publicKeyPointer` in both contracts and their tests + call-sites. Logged as A-001 in `docs/amendments.md` at the point of change. After rename, all four PQC accounts share naming (`MlDsaAccount`, `FalconAccount`, `MlDsaEthAccount`) while `EcdsaAccount` remains a pure `SimpleAccount` subclass with no custom pk storage.

Production paths use `crypto.getRandomValues(32)` for ζ / rnd (hedged sign; Python `deterministic=False`).

### `test/signers/mldsa-encoding.ts` (refactored)
```ts
export interface XofReader {
  readonly id: "shake128" | "shake256" | "keccak-prg";  // named discriminant (M-3)
  xof(length: number): Uint8Array;                       // stateful; may be called repeatedly
}
export type XofFactory = (seed: Uint8Array) => XofReader; // fresh reader per call-site
export function preparePublicKeyForDeployment(rawPk: Uint8Array, xofFactory: XofFactory): Uint8Array;
```

XOF is required (no default). NIST adapters: `shake128XofFactory` / `shake256XofFactory` wrap `shake{128,256}.create().update(seed)` with `id: "shake{128,256}"`. ETH adapter: `(seed) => { const p = createKeccakPrg(seed); p.flip(); return { id: "keccak-prg", xof: (n) => p.extract(n) }; }`. Every call-site constructs its own reader — AC-A-1 HIGH enforced by grep (no module-level factory references; no `let _xof` / `var _xof`). The `id` discriminant surfaces in `assertBytesEqual` divergence messages ("first divergent byte at offset N (factory=keccak-prg)") so bisecting XOF-swap bugs has a grep-friendly anchor per maintainer review M-3. Factory-vs-one-shot rationale lives in DD-10. Notable migration site: `tr = xofFactory(ρ ∥ t1_encoded).xof(64)` (formerly hardcoded `shake256` at `mldsa-encoding.ts:103`). `ml-dsa.ts` NIST caller updates to `preparePublicKeyForDeployment(pk, shake256XofFactory)`; AC-D-1 enforces byte-identical NIST regression. Refactor commit logged as a Rule 3 amendment in `docs/amendments.md`.

**Module-header Delta requirement (M-3):** `ml-dsa-eth.ts` and `ml-dsa-eth.kat-internal.ts` each start with a JSDoc block including a `@delta-from-ml-dsa` section enumerating byte-level differences from the NIST path — XOF (Keccak-PRG vs SHAKE-128/256), noble-fork scope (keygen+sign both XOF-parameterized), pk-transform factory argument, `ctx` handling, signature layout. Mandatory for onboarding — makes the mutual near-identity + byte-incompatibility explicit where a maintainer first lands.

### `test/signers/keccak-prg.ts` (new)

Port of ZKNox's `Keccak256PRNG`. Three-phase stateful construction: `inject` (absorb arbitrary input chunks into a buffer) → `flip` once (`state = keccak256(buffer)`) → `extract(n)` repeatedly (blocks of `keccak256(state ‖ counter_be_u64)`, counter starts at 0 and increments per full 32 B block; out-buffer-pos retains partial blocks so sub-32 B extracts don't advance the counter until the block is drained).

```ts
export interface KeccakPrg {
  inject(data: Uint8Array): void;      // absorb; throws if called after flip
  flip(): void;                         // finalize; throws if called twice
  extract(length: number): Uint8Array;  // stream; throws if called before flip
  update(data: Uint8Array): void;       // alias for inject (noble SHAKE parity)
  read(length: number): Uint8Array;     // alias for extract  (noble SHAKE parity)
}
export function createKeccakPrg(seed?: Uint8Array): KeccakPrg;
// Optional ctor seed is equivalent to prg.inject(seed). Not auto-flipped —
// caller calls flip() after any additional inject(), mirroring Python.
```

Stateful because rejection sampling in ExpandA/ExpandS/ExpandMask/SampleInBall pulls bytes in multiple chunks on the same seeded instance with unpredictable total length — a stateless `(seed, outLen) => bytes` signature can't express that without over-allocation or divergence from ZKNox's construction (see DD-10). Byte-compatibility invariants are enumerated as G0 KAT cases in `keccak-prg/vectors.json`; conformance is asserted there, not here.

## Security

**Tier: Low (research tool, local Hardhat only, no PII / finance / deployment).** Cryptographic correctness is covered by AC-A-1 / AC-D-2 / AC-NFR-1 / AC-NFR-2 under Testing, not here.

- **Submodule pinning** — `.gitmodules` records ETHDILITHIUM SHA; fixture-gen CLI refuses to run when HEAD differs (AC-NFR-4). Protects against silent Python-ref drift.
- **Dev-oracle isolation** — no Python interpreter invoked by any shipped test path. Only `scripts/generate-kat-fixtures.ts` spawns Python, and it's not part of `npm test`. NFR-3 guarded by code review (explicit user deferral).
- **No secrets** — KAT fixtures include test secret keys by design (they are research fixtures, not production keys). README attribution (AC-U-5) states this explicitly.
- **Input validation** — signer throws structured errors on malformed inputs (AC-U-1 codes); account contract reverts on malformed signatures (AC-A-2). No trust is placed in user input at crypto layer.
