# Brief: ML-DSA-ETH — Keccak-Based ML-DSA for ERC-4337

## Problem

Our pqc-4337 benchmark (completed as a prior feature) measured ML-DSA-44 validateUserOp at ~8.3M gas per call on Hardhat's EntryPoint simulator — about 110× ECDSA's cost. ZKNox has published a Keccak-based ML-DSA variant (`ZKNOX_ethdilithium.sol`) that claims ~4.9M gas by swapping SHAKE-128/256 for Keccak-256 (Ethereum's native precompile-adjacent hash) and pre-expanding the public key so the verifier skips the A_hat matrix derivation. Our current benchmark does not include this variant, so a reader evaluating PQC options for account abstraction cannot see the Keccak trade-off side-by-side with the NIST-spec ML-DSA, Falcon, or ECDSA.

Worse, there is no JavaScript signer for ETHDilithium/MLDSAETH in the open-source ecosystem. ZKNox ships a Python reference (`sign_cli.py --version MLDSAETH`) that works with their verifier, but a hardware wallet, browser extension, or node-based bundler client has no equivalent path. Without a JS signer, the verifier cannot be exercised from the Node.js + viem + hardhat test stack used by this benchmark and the broader Ethereum tooling ecosystem.

## Vision

Add ETHDilithium (hereafter "ML-DSA-ETH") as the 4th scheme alongside ECDSA, Falcon, and ML-DSA in the existing benchmark framework. Port `@noble/post-quantum`'s `ml_dsa44` JS implementation to a Keccak-256-backed variant that is byte-compatible with `ZKNOX_ethdilithium.sol` — same public-key reshape, same challenge derivation, same PRG calls. Use ZKNox's Python reference as a development-time oracle (never shipped in the final codebase) to generate known-answer test vectors during bring-up. Wire an `MlDsaEthAccount` SimpleAccount override that calls the ZKNox verifier, add full acceptance/rejection + failure-class test coverage matching the other schemes, and extend the benchmark report so the 4-way comparison is a one-command reproduction.

## Users

- **PQC researchers / AA protocol designers**: Run `make bench` (or equivalent) and read a report that places ML-DSA-ETH next to the three existing schemes. Goal: evaluate whether Keccak-based ML-DSA is a viable middle ground between Falcon's size-advantage and NIST ML-DSA's spec-compliance. Frustration: ZKNox's 4.9M figure is only in their README — no independent reproduction exists.
- **ERC-4337 wallet engineers**: Want a reference JS signer they can vendor into a browser extension or node client. Current pain: `@noble/post-quantum` covers NIST ML-DSA but not the Keccak variant; rolling your own without a known-correct oracle is risky.
- **ZKNox team**: Indirect beneficiary. An independent port validates their design and stresses byte-level compatibility with their Solidity verifier.

## Success Metrics

- **Primary (functional)**: 100% of userOps signed by our JS ML-DSA-ETH signer verify successfully on-chain via `ZKNOX_ethdilithium.sol`. Measured by acceptance tests run through our Hardhat + viem stack.
- **Primary (correctness)**: ≥10 known-answer test vectors generated by ZKNox's Python reference (`sign_cli.py --version MLDSAETH`) verify byte-for-byte against our JS signer output. Measured at test-vector-generation story.
- **Secondary (informational, no threshold)**: Average gas cost of `MlDsaEthAccount.validateUserOp` recorded in the benchmark report next to the other three schemes. Reader can form their own conclusion about the 4.9M figure vs our measurement.
- **Scope hygiene**: Zero Python files in the final tree except under `ETHDILITHIUM/` submodule (dev-time oracle, git-ignored from shipped package).

## Constraints

- **Byte-compatibility**: Any deviation in challenge derivation, PK layout, or signature encoding breaks on-chain verification. The JS signer must produce output that `ZKNOX_ethdilithium.sol` accepts — no wiggle room.
- **Dev-oracle pin**: ZKNox's `ETHDILITHIUM/` submodule must be pinned to a specific commit for reproducibility of generated test vectors.
- **No new Solidity compiler**: hardhat.config's current `solc` version must compile ZKNox's contracts. If not, this is a stop-and-escalate moment (likely triggers a DD).
- **Tooling continuity**: Same stack as pqc-4337 — hardhat v3, viem 2.43+, `@noble/post-quantum` 0.6.1 as port basis, node:test runner.
- **NIST ML-DSA-44 parameters**: k=4, l=4, η=2, γ_1=2^17, γ_2=95232, τ=39, ω=80, β=78 (unchanged; only the XOF swaps).
- **No modification to archived pqc-4337 artifacts**: `docs/.archive-pqc-4337-20260416-115932/` is frozen history.
- **Research ethics**: Credit ZKNox in the final README; their design, our independent port.

### Critical delta between ML-DSA and ML-DSA-ETH

From ZKNox's source:

1. **XOF replacement**: Every call to SHAKE-128 (for ExpandA) and SHAKE-256 (for ExpandS, ExpandMask, H, G, SampleInBall) is replaced by `Keccak256PRNG`. The PRNG is a deterministic stream-cipher-style construction over Keccak-256 — takes a seed, emits unbounded bytes.
2. **Public key reshape**: Standard ML-DSA pk = `(ρ, t1)`; verifier re-derives A_hat from ρ at every verify. ETHDilithium pk = `(A_hat_precomputed, tr, t1_new)` where A_hat is the full expanded NTT-domain matrix and `tr` is the precomputed `H(ρ ∥ t1)` challenge prefix. This saves the verifier ~k·l Keccak-expansion calls per verify.
3. **Signer implications**: The JS signer must also use the reshaped form — it signs with the same `tr` that the verifier will see, not by re-hashing ρ and t1 every time. This means the signer's keygen output is `(sk, pk_eth)` where pk_eth is already transformed.

### Existing pk_for_eth pipeline (already written for pqc-4337)

`test/signers/mldsa-encoding.ts#preparePublicKeyForDeployment` (written for our existing NIST-variant `MlDsaAccount` against ZKNox's `ZKNOX_dilithium.sol`) already implements the pk reshape:

1. `decodePublicKey(pk)` → parses raw 1,312-byte NIST pk into `(ρ, t1, tr)` where `tr = shake256(pk, dkLen=64)`.
2. `recoverAhat(ρ, 4, 4)` → rejection-samples A_hat matrix from ρ via SHAKE-128.
3. `transformT1Poly(poly)` → Power2Round `<<D=13` shift + forward NTT per t1 polynomial.
4. `compactModule256(..., 32)` → packs coefficients into 32-bit compact uint256 arrays.
5. ABI-encode `(aHatEncoded, tr, t1Encoded)` as the SSTORE2 payload consumed by `ZKNOX_dilithium.setKey()` → `_readPubKey()`.

ML-DSA-ETH differs from this pipeline only in the XOF used at three call-sites:
- `rejectionSamplePoly` (line 45): `shake128.create()` → Keccak-256 PRG.
- `decodePublicKey` (line 103): `shake256(pk, 64)` → Keccak-256 PRG.
- Signer-side ExpandS / ExpandMask / SampleInBall / H / G (not yet in this tree — will live in `test/signers/ml-dsa-eth.ts`): SHAKE-256 → Keccak-256 PRG.

The NTT step, Power2Round shift, compact encoding, and ABI layout are identical across both variants — they operate on coefficients/bytes and carry no XOF. This collapses what initially looked like "port the whole pk reshape" into "inject a different XOF into the existing reshape." See DD-2 for the locked mechanic vs deferred implementation strategy.
